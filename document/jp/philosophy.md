# 設計思想

LazyNixは、再現可能な開発環境を手に入れるために新しいプログラミング言語を学ぶ必要はないはずだ、という考えから生まれました。この章では、LazyNixを形作る設計判断と、それに伴うトレードオフを説明します。

## 課題

開発環境のセットアップは、プロジェクトに参加する際の最もフラストレーションが溜まる作業のひとつです。READMEには「Python 3.12、Node 18、PostgreSQLをインストールしてください」と書かれていますが、どのマイナーバージョンか、どのビルドフラグか、どんなプラットフォーム固有の注意点があるかは、ほとんど書かれていません。結果として、おなじみの儀式が始まります。新しいチームメンバーがアプリケーションコードを一行も書く前に、環境の差異をデバッグするのに何時間も費やすのです。

Nixはこの問題を根本的に解決します。Nix flakeはすべての依存関係を暗号学的な精度で宣言でき、すべての開発者がビット単位で同一の環境を手に入れることを保証します。しかし、Nixには別の問題があります。それ自体の学習コストです。Nix言語は遅延評価の純粋関数型言語であり、見慣れない構文を持っています。「全員が同じPythonを使えればいい」というチームにとって、Nixの学習は過剰な投資に感じられます。

LazyNixはこのギャップを埋めます。

## YAMLのみによる設定

LazyNixの核となる設計判断は、ユーザーがNixを一切書かないことです。代わりに、必要なものを宣言する `lazynix.yaml` を書きます:

```yaml
devShell:
  package:
    stable:
      - python312
      - uv
    unstable: []
```

LazyNixはこれを裏側で有効な `flake.nix` に変換します。ユーザーはNixの式、属性セット、nixpkgsのモジュールシステムを理解する必要がありません。必要なパッケージの名前さえ知っていればよく、それは [search.nixos.org](https://search.nixos.org/packages) で調べられます。

これは意図的なトレードオフです。LazyNixはNixの完全な表現力を犠牲にし、代わりにどの開発者でも数秒で読み書きできるインターフェースを提供します。YAMLファイルは自明です。Nix式は、初めての人にとってはそうではありません。

### 既存ツールの活用

LazyNixは新しいエコシステムを発明しません。すでに存在するツールを活用します:

- **nixpkgs** がパッケージを提供します。LazyNixは独自のパッケージリポジトリを持ちません。
- **Nix flakes** が再現性を保証します。LazyNixは標準的なflakeを生成します。
- **nix develop** がシェル環境を提供します。LazyNixはそれを直接呼び出します。

既存のインフラの上に構築し、それを置き換えないことで、LazyNixは薄い変換層にとどまります。nixpkgsに新しいパッケージが追加されれば、LazyNix自体を変更することなく、LazyNixのユーザーはすぐにそのパッケージにアクセスできます。

### インターフェースの簡略化

LazyNixは、開発者が理解しなければならない領域を縮小します。それぞれのアプローチで開発者が知る必要があることを比較してみましょう:

**LazyNixなし（純粋なNix）:**
- Nix言語の構文
- `flake.nix` の inputs と outputs の書き方
- `mkShell` 関数とその引数
- `nixpkgs` の属性パスの仕組み
- `flake.lock` によるバージョン固定の方法

**LazyNixあり:**
- YAMLの構文
- search.nixos.org でのパッケージ名

それ以外はすべてLazyNixのコード生成が処理します。

## DevShellのみ

LazyNixがやることはひとつ、開発シェル環境の作成です。パッケージのビルドはしません。NixOSのシステム設定は定義しません。デプロイインフラの管理もしません。

この制約は意図的なものです。Nixは開発ツールから本番サーバーまであらゆるものを管理できる広大なエコシステムです。しかし、ほとんどのチームにとって最も投資対効果が高いユースケースは開発環境です。すべての開発者を同じツールチェーンに揃えることで、環境差異に起因するバグと時間の浪費をまるごと排除できます。

この単一のユースケースに集中することで、LazyNixは設定の表面積を小さく保ち、エラーメッセージを具体的にできます。何か問題が起きたとき、原因はほぼ「このパッケージ名が存在しない」か「このパッケージがお使いのプラットフォームでは利用できない」のどちらかです。そして、それは `lnix lint` がユーザーに届く前にキャッチする問題です。

## 脱出経路

LazyNixは標準的な `flake.nix` ファイルを生成します。これは実装の詳細ではなく、設計上の保証です。いつでもチームはLazyNixの使用をやめて、生成された `flake.nix` を直接編集し続けることができます。

移行は2ステップです:

1. `lazynix.yaml` を削除する
2. 今後は `flake.nix` を手動で編集する

再配線も、エクスポート処理も、データの損失もありません。生成された `flake.nix` は、LazyNixがインストールされていなくても動作する、完全で独立したNix flakeです。

この脱出経路が重要なのは、チームは成長するからです。「Pythonとリンターがあればいい」で始まったプロジェクトも、いずれカスタムのNixオーバーレイ、クロスコンパイル、あるいはより大きなNixインフラとの統合が必要になるかもしれません。その日が来たとき、LazyNixは制約になるのではなく、穏やかに道を譲ります。

ここでの設計思想は、**ツールはユーザーを閉じ込めてはならない** ということです。LazyNixはNixへの参入障壁を下げます。ユーザーがその障壁を超えて成長したら、ペナルティなくフルNixへ卒業できます。

## LazyNixが目指さないもの

LazyNixのスコープを明確にするために、明示的に目指していないものを示します:

- **Nixの代替品。** LazyNixはNixを生成します。実行時にNixを置き換えたり抽象化したりはしません。
- **パッケージマネージャ。** LazyNixはパッケージをインストールしません。Nixがインストールします。LazyNixはNixにどのパッケージを含めるか指示するだけです。
- **ビルドシステム。** LazyNixはコードをコンパイルしたり、成果物を生成したり、ビルドパイプラインを管理したりしません。
- **システム設定ツール。** LazyNixはNixOSモジュール、Home Manager、システムレベルの設定を管理しません。

LazyNixが何でないかを明確にすることで、ユーザーはいつLazyNixを使い、いつ別のツールに手を伸ばすべきか、十分な情報に基づいて判断できます。

## まとめ

LazyNixは3つの原則の上に構築されています:

1. **宣言による設定。** ユーザーは必要なものをYAMLで宣言する。LazyNixが「どうやるか」を処理する。
2. **焦点を絞ったスコープ。** 開発シェル環境のみ。ひとつのことをうまくやる。
3. **ロックインなし。** 生成された `flake.nix` が脱出経路。ユーザーはいつでも離れられる。

これらの原則が、LazyNixをシンプルで、予測可能で、その限界に対して正直に保っています。
